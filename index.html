What is Ponytail?

Ponytail is a set of skills and rules for AI coding agents that enforce the laziest solution that actually works. It channels a senior developer who has seen every over-engineered codebase and been paged at 3am for one.

Before writing any code, the agent walks a decision ladder:

  1. Does this need to exist? Speculative need = skip it. (YAGNI)
  2. Stdlib does it? Use it.
  3. Native platform feature? Use it.
  4. Already-installed dependency? Use it.
  5. Can it be one line? One line.
  6. Only then: write the minimum code that works.

Lazy, not negligent: input validation, error handling, security, and accessibility are never on the chopping block.

Quick Start

System-wide install:

git clone https://github.com/pavnxet/Mimocode-ponytail.git
cd Mimocode-ponytail
./install.sh    # Linux/Mac
install.bat     # Windows

Per-project (no install):

git clone https://github.com/pavnxet/Mimocode-ponytail.git
cd Mimocode-ponytail
mimocode

mimocode reads AGENTS.md at repo root and loads skills from skills/ automatically.

Skills

ponytail

Lazy senior dev mode. The simplest solution that works. Supports lite / full / ultra intensity levels.

ponytail-review

Review diffs for over-engineering. One line per finding: location, what to cut, what replaces it.

ponytail-audit

Whole-repo audit for bloat. Ranked list of what to delete, simplify, or replace with stdlib/native.

ponytail-debt

Harvest ponytail: comments into a debt ledger. Track deferred shortcuts before they rot.

ponytail-help

Quick reference card for all modes, skills, and commands.

Intensity Levels

LevelTriggerWhat changes
lite/ponytail liteBuild what's asked, name the lazier alternative in one line. User picks.
full/ponytailThe ladder enforced. Stdlib and native first. Shortest diff, shortest explanation. Default.
ultra/ponytail ultraYAGNI extremist. Deletion before addition. Challenges requirements before building.

Level persists until changed or session end. Say "stop ponytail" or "normal mode" to deactivate.

Example: "Add a cache for these API responses."

LevelResponse
lite"Done, cache added. FYI: functools.lru_cache covers this in one line if you'd rather not own a cache class."
full"@lru_cache(maxsize=1000) on the fetch function. Skipped custom cache class, add when lru_cache measurably falls short."
ultra"No cache until a profiler says so. When it does: @lru_cache. A hand-rolled TTL cache class is a bug farm with a hit rate."

Without Ponytail

npm install flatpickr + 30-line wrapper component, two useEffect hooks, cleanup function, CSS import.

Email Validation

Without Ponytail

27-line EmailValidator class with regex that still rejects valid addresses and accepts invalid ones.

With Ponytail

# ponytail: good enough, real validation is sending the mail
"@" in email and "." in email.split("@")[-1]

Sorting

Without Ponytail

24-line hand-rolled quicksort that allocates three arrays per recursion level and blows the stack on large inputs.

With Ponytail

// ponytail: this exists
numbers.sort((a, b) => a - b)

Caching

Without Ponytail

120-line TTLCache class with threading, LRU eviction, hit/miss stats — for a problem nobody has measured yet.

With Ponytail

# ponytail: stdlib covers this
from functools import lru_cache

@lru_cache(maxsize=1000)
def fetch(key): ...

API Endpoint

Without Ponytail

5 files, 3 classes, custom exception, dependency-injection chain — wrapping one database call.

With Ponytail

class UserOut(BaseModel):
    id: int
    name: str
    email: str

@app.get("/users/{user_id}", response_model=UserOut)
def get_user(user_id: int, db = Depends(get_db)):
    user = db.get(User, user_id)
    if not user:
        raise HTTPException(404)
    return user

Agent Portability

Ponytail works across multiple AI agent hosts. The core skills live in skills/; host-specific files are thin adapters.

HostFilesNotes
mimocodeAGENTS.md, skills/, commands/Always-on rules from AGENTS.md, skills loaded on demand.
Generic agentsAGENTS.md or skills/*/SKILL.mdCopy the rule file or load skill files directly.

Configuration

Default mode is full. Override it:

Environment variable

export PONYTAIL_DEFAULT_MODE=ultra  # lite | full | ultra | off

Config file

~/.config/ponytail/config.json (Windows: %APPDATA%\ponytail\config.json)

{ "defaultMode": "lite" }

Resolution: env var > config file > full.

The Rules

  • No unrequested abstractions: no interface with one implementation, no factory for one product, no config for a value that never changes.
  • No boilerplate, no scaffolding "for later", later can scaffold for itself.
  • Deletion over addition. Boring over clever, clever is what someone decodes at 3am.
  • Fewest files possible. Shortest working diff wins.
  • Complex request? Ship the lazy version and question it in the same response.
  • Mark deliberate simplifications with a ponytail: comment naming the ceiling and upgrade path.

What is never simplified away

  • Input validation at trust boundaries
  • Error handling that prevents data loss
  • Security measures
  • Accessibility basics
  • Anything explicitly requested

FAQ

Does it need a config file?

No. Optional ~/.config/ponytail/config.json or PONYTAIL_DEFAULT_MODE env var can set the default level, but nothing is required.

What if I really need the 120-line cache class?

You don't. Insist anyway and he'll build it. Slowly. Correctly. While looking at you.

Does it scale?

The code you never wrote scales infinitely. Zero bugs, zero CVEs, 100% uptime since forever.

Why "ponytail"?

You know exactly why.
